Explore Writeup (HackTheBox)

Overview

Machine: https://www.hackthebox.eu/home/machines/profile/356 Explore is an easy HackTheBox Android device created by bertolis (https://www.hackthebox.eu/home/users/profile/27897). We begin with finding an Android ES File Explorer vulnerability, in which user data are exposed, making us able to get user’s credentials from a screenshot on the device. From there we can get access on the device via ssh. We then abuse the fact that Android Debug Bridge (ADB) is enabled on the device in order to get a privileged shell.

Investigating port 2222

From the output of nmap we can observe that this port is ssh. More specifically, the version enumeration flag of nmap informs us that this is SSH-2.0-SSH Server - Banana Studio. This was interesting so googling for Banana Studio ssh, reveals us that this is an Android application, which means we have to deal with an Android box. Nevertheless, this is no different than any other ssh client, which means it’s a dead end for now since we have no valid username or password to work with.

Investigating port 5555

Moving to port 5555, it states to be filtered. As stated on the nmap page, the filtering could be from a dedicated firewall device, router rules, or host-based firewall software. Further googling about how port 5555 is related to Android devices, brought me to this article. It appears that multiple Android devices are getting shipped with Android Debug Bridge (ADB) enabled. ADB allows developers to communicate with a device remotely without the need to authenticate in order to run a variety of commands, including running shell commands as the highest prileged user, root. Since this appears to be the case it is very possible that by connecting to this port, will give us root access to the device. To connect to this port we will need android-tools-adb which do not come pre-installed with Kali Linux. To install them we run:

1

apt-get install android-tools-adb

At this point we need to connect to this port with adb. To do this we run:

1

adb connect 10.10.10.247:5555

This command hangs for a while and then eventually times out. It seems that this port is indeed filtered which means we won’t be able connect to it. It is something to keep in mind since it’s a way to get root on the device according to the article, however it is a dead end for the time being.

Enumerating port 59777

While searching online for information about port 42135 brought no results, searching for port 59777 instantly brought results. The result of the search was this article which describes a vulnerability where Android ES File Explorer exposes user data through the open port 59777. This vulnerability, discovered by a security researcher named Baptiste Robert allows anyone connected to the same local network to remotely get a file from the device using ES File Explorer. Moreover, a POC python script can also be found here. It seems this is our way of listing the device’s files.

Escalating Privileges to root

After getting a shell on the device, we see that we have limited rights to do pretty much anything. Searching for config files, password files, pictures or any other useful file brought no results. At this point we remember that the nmap scan showed us that the device was listening on port 5555, something that we found out to be a vulnerability that could lead an attacker obtaining root access to the device. At that point we could not do anything about it since it was a filtered port, but now we are inside the device. At first we want to see if the device is indeed listening on this port:

1
2

netstat -an | grep LISTEN | grep 5555                                     
tcp6       0      0 :::5555                 :::*                    LISTEN

We confirm that the device is listening on port 5555 which means we can again try to connect to it with adb. The problem here though, is that android-tools-adb is not installed on the device. So in order to interact with this port we need to do it from our machine, which has the tools we need installed. Did we not try that before though? The answer is no, because now we have valid ssh credentials and access on the device, which means we can now use port forwarding, in order to connect to port 5555, bypassing any firewall rule that was previously preventing us to do so. To port foward we use:

1

ssh -L 6666:localhost:5555 kristi@10.10.10.247 -p 2222 -fN

Flags explained:

  • L: This specifies that we are going to use a Local Port Forwarding. What we are going to do in this case is make all traffic sent to port 6666 on our local host to be forwarded to port 5555 on the target device via the stablished ssh connection.
  • -fN: This is a combined switch that does two things: -f backgrounds the shell immediately so that we have our own terminal back. -N tells SSH that it doesn’t need to execute any commands, only set up the connection.

Of cource since we are going to make an ssh connection as the Kristi user we will get asked for the password we previously got. Hitting enter will background the command and will seem like it did nothing, however the connection was made and is active as we can see below:

What we have to do now, is repeat the steps we made at the start with adb, however we will now connect on port 6666 on our machine! What is going to happen, is that this connection will be forwarded on port 5555 of the target machine, letting us right in. We start by connecting to port 6666 with adb:

1

adb connect localhost:6666

The message states that we are connected, so the next step would be to list our connections to confirm it:

1

adb devices -l

We see that indeed we are connected to localhost:6666 and we get some info about the device we are connected to. We are now able to execute the last step, which is launching a shell on that device. To do this we use:

1

adb -s localhost:6666 shell

We now have a shell on the device. While the whoami command shows us that we are not root, the shell user of adb is meant for developers and has more privileges than other users, including the ability to change to the superuser (root). There is a great article to read about the shell user permissions listed on the Resources section at the end of this post:

1

su root

Exploring Ghost Framework

Searching around the Internet, i came across a very intersting tool, called Ghost Framework, an “Android post-exploitation framework that exploits the Android Debug Bridge to remotely access an Android device”. First we need to dowload it with:

1

pip3 install git+https://github.com/EntySec/Ghost

The we can run it with:

1

ghost

This will open the Ghost Framework, where we can enter help to see our available options:

All we need to do is connect to the device. We have already port forwarded our traffic from our local port 6666 to the port 5555 of the target, so we are going to connect to localhost:6666 with:

1

connect localhost:6666

We can then also type help to see our options:

To get root all we need to do is type shell to get into a shell on the device as the user shell and then su root to get root, the same way we did before.

While getting root on this particular device with Ghost Framwork is not needed, it can do many more things, that may be very useful in a real life scenario. With Ghost Framework we can:

  • Show device activity information.
  • Show device battery state.
  • Show device network information.
  • Show device system information.
  • Control device wifi service.
  • Click the specified x and y axis. (For example click the show password while a user is on a login screen and then take a screenshot of the password)
  • Control device keyboard.
  • Press/Simulate keypress on target device.
  • Open a URL on the device.
  • Control device screen.
  • Take a screenshot from the device.
  • Open a shell on the target device.
  • Type a specified text on the device.
  • Upload a local file from our machine to the target device.
  • Download files from the target device.
  • Show the Contacts that are saved on the device.

All in all, Ghost Framework seems to be another simple yet powerful tool to use for Android Post-Exploitation.
